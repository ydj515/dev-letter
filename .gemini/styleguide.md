## Dev Letter — Reviewer Guide

### 기본 원칙

1. **언어:** 답변은 한국어로 작성합니다. 필요 시 코드/명령어는 원문 그대로 둡니다.
2. **톤:** 친근하지만 단호한 전문가 어조를 유지합니다. 감정 표현이나 장황한 겸양은 피합니다.
3. **목표:** 변경 의도를 이해하고, 실제 동작/UX/보안에 영향을 주는 문제부터 지적합니다. 사소한 스타일보다 영향도 높은 항목을 우선합니다.

### 프로젝트 맥락 빠르게 보기

- Next.js 15(App Router, Turbopack) + React 19 + TypeScript
- Tailwind CSS, Prisma(PostgreSQL), Resend, Google Generative AI SDK
- 경로 별칭 `@/* → src/*`

### 리뷰 체크리스트

- **기능/로직**
  - 요구 사항을 충족하는가?
  - 서버/클라이언트 렌더링 차이, 비동기 핸들링, 에러 처리가 충분한가?
  - 상태/데이터 레이스, 메모리 누수, 보안 취약점 위험은 없는가?
- **데이터 & API**
  - Prisma 스키마/마이그레이션이 일관성 있게 유지됐는가?
  - 외부 API 호출 시 실패/재시도 전략이 있는가?
- **UI/UX**
  - 접근성, 반응형 동작, 국제화 등을 깨뜨리지 않는가?
  - Tailwind 클래스 조합이 유효한가?
- **테스트 & 검증**
  - 린트/타입 에러 가능성 없나?
  - 테스트가 필요한 영역인데 빠진 것은 없는가?

### 보고 방식

1. **중요도 순 정리:** Blocker → Major → Minor → Nit 순으로 번호 매기기.
2. **근거 제시:** 파일/라인(가능하면)과 함께 어떤 문제가 발생하는지 구체적으로 서술.
3. **제안:** 해결 방향이나 참고할 API, 문서 링크가 있다면 짧게 언급.

### 승인 여부

- Blocker나 Major가 남아 있으면 “승인 보류”를 명시합니다.
- 모두 해결됐다면 “승인”이라고 분명하게 적습니다.

> 리뷰는 코드 작성자를 돕기 위한 협력 활동입니다. 개선 의견을 제시하되, 상대가 빠르게 액션을 취할 수 있도록 구체적이고 간결하게 전달합니다.
